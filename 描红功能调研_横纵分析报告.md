## 一、一句话定义

**描红的"笔顺贴合字体"问题**,本质是:在数字屏幕上,让"引导虚线(浅色字或虚线 path)"和"字体本身的轮廓"在视觉上完全重合,并且能实时判断用户笔触"是否对得上"这一标准字形 —— 它既是渲染问题,也是数据问题,更是算法问题。

## 二、纵向分析:从纸笔到 AI,描红技术的 16 年演进

### 2.1 纸笔时代(2000-2010):千年传统的最后阵地

描红(古人称"描朱"或"顺朱")的历史远比我们想象的久远。明清时期已经规模化,学童最常用的描朱教材是《上大人》,这一形态在中国延续了上千年。

这一时期儿童练字产品的核心设计原则,被后来所有数字化产品继承:

1. **田字格 / 米字格的格子定位系统**:把一个字的几何中心、笔画起止点固定在方格内。至今,洪恩、宝宝巴士、有道的写字笔产品里,都能看到田字格的身影。
2. **红色楷书范字 + 黑色书写区域**的对照关系:这条从《上大人》传承下来的视觉范式,被 2024 年腾讯云发布的《WPS 做描红:数字化练字的新纪元》原样搬到了 WPS 文档模板。
3. **笔画顺序的展示与重复**:"8 句口诀"式的笔画口诀,无论在田字格贴纸还是 2010 年代的 Flash 课件里,都是核心教学载体。

**这一时代的局限**:纸笔无法自动判断孩子写得好不好,反馈完全依赖家长和老师;笔画数据不标准,不同字帖对"毕"字的第一笔是"横"还是"竖勾"有分歧。

### 2.2 电子化起步(2010-2015):iPad 浪潮与第一批儿童练字 app

2010 年,iPad 横空出世。这一年的儿童教育 app 市场有几个关键事件:

**1. 洪恩的早期布局**
洪恩(北京洪恩教育科技股份有限公司)成立于 2003 年,创始人池宇峰。2010 年时,洪恩已经覆盖 1.5 万所幼儿园、300 多家培训学校,产品出口至 10 多个国家 —— 这是行业内最早完成"线下 → 数字化"基础设施的玩家。但 2010 年的洪恩主要还是**光盘 + 点读笔**的形态,真正发力"识字"是 2016-2018 年的事。

**2. 悟空识字:西游 IP + Flash 动画的初代范本**
悟空识字 2010 年上线,最早是基于 Cocos2d-x 框架、用 Flash 动画承载的桌面端产品,后迁移到 iPad。它把 1200 个常用汉字、5000 多个词语套进了《西游记》剧情。从技术角度看,这一代产品的"笔顺"是**预渲染好的 Flash 动画**或 GIF 序列,无法做交互 —— 孩子只能"看动画",不能"自己写"。

**3. 笔顺展示的技术实现(2010-2015)**
这一时期的"笔顺"本质上是两种技术:

- **预渲染动画**:Flash / GIF 视频,做不了交互,只能"看"。
- **TrueType 字体反推**:CSDN 上 2010 年前后的技术博客展示了用 Windows GDI+ Path 技术从 TrueType 字体数据反推笔顺的方案 —— 这种方案被后来的部分国产 app 沿用,但**精度极差**(因为 TrueType 字体本身不带"先写哪一笔"的语义信息,反推出来的顺序常常和标准笔顺相反)。

**这就是为什么"毕"字会被写错** —— 在没有权威开源笔顺数据的年代,每个 app 各说各话,孩子学到的东西可能本身就是错的。

### 2.3 标准化与开源(2015-2020):Make Me a Hanzi 与 Hanzi Writer 横空出世

2015 年前后,儿童练字/描红产品发生了一场静悄悄但影响深远的**基础设施革命** —— 这要从一个叫 skishore 的开发者说起。

**1. Make Me a Hanzi(2014-2015):开源汉字数据的奠基之作**

GitHub 仓库 https://github.com/skishore/makemeahanzi 显示,该项目由 skishore 主导,数据来源于台湾**文鼎科技(Arphic Technology)**于 1999 年以宽松许可协议发布的"Arphic PL KaitiM GB"和"Arphic PL UKai"两款楷书字体。skishore 把这些字体的笔画、字形结构(IDC 描述序列)、拼音、释义、部首等数据手工整理成两个文件:

- `dictionary.txt`:9000+ 汉字的字典数据(释义、拼音、IDS 分解、字源)
- `graphics.txt`:每个汉字的 SVG 路径,按**标准笔顺**排列(1024×1024 坐标系)

**这个项目的革命性在于**:
- 首次提供了**可机读的、经过验证的 9000+ 汉字标准笔顺 SVG 路径**;
- 采用 Arphic 的**楷书**(而非宋体),更适合儿童学习;
- 数据文件是**逐行 JSON**,开发者可以直接解析;
- 截至 2026 年,该项目已获 **2.5k stars / 590 forks**。

GitHub 主页明确列出了它的下游用户:**Hanzi Writer JS 库**、**HanDeDict 中德词典**、**Hanping Chinese Dictionary**(iOS / Android)、**繁体字帖**iOS app 等,形成了一个完整的开源汉字数据生态。

**2. Hanzi Writer(2015+):前端笔顺动画的事实标准**

Hanzi Writer 仓库由 chanind 创建,目前已积累 **4.8k stars / 708 forks,258 次 commit,58 个 release**(最新 v3.7.3,2025 年 9 月发布)。它本质上是一个**基于 Make Me a Hanzi 数据的渲染层**:

- 用 SVG 渲染汉字
- 用 SVG `<path>` 配合 `stroke-dasharray` 实现**逐笔动画**
- 内置**手写测验**功能(quiz mode)—— 可以判断用户的笔画顺序、起止点、覆盖区域是否正确
- 库大小仅 **35 KB(min 版 9 KB)**,可嵌入 Web、iOS、Android、小程序

根据 CSDN 2019 年的介绍和掘金 2024 年的报道,Hanzi Writer 已成为"全球中文学习开发者**人手一份的笔顺库**"。

**3. 微信小程序生态的爆发(2017-2020)**

2017 年微信小程序正式上线后,儿童教育赛道快速迁移:

- 2017 年,**北京语言大学**信息科学学院安维华、刘颖滨老师团队发布"汉字笔画、笔顺大测试"小程序,提供**笔划美化、笔顺判别、书写判别**三大功能 —— 这是学术界第一次把"AI 笔顺判别"做成了消费级小程序。
- 2019 年,`hanzi-writer-miniprogram` 微信小程序组件发布,成为小程序描红的事实路径。但 2024 年 2 月 20 日被 archived,反映**微信小程序生态的快速变化**(基础库升级、Canvas API 重构、jsdelivr CDN 备案撤销等多重原因)。

**关键转折点**:开源数据(Make Me a Hanzi / hanzi-writer-data)让**独立开发者和中小团队也能做出"标准笔顺"的描红产品**。在这之前,大厂的笔顺数据是私有资产;之后,它成了**公共基础设施**。这就是为什么 2020 年之后,微信小程序里"练字 / 描红"赛道会出现**指数级增长** —— 技术门槛被 Make Me a Hanzi 打平了。

### 2.4 AI 化与精准描红(2020-2026):从"能写"到"写对"

2020 年起,儿童练字产品进入**AI 化阶段**。这一阶段的关键变化,是从"展示正确笔顺"升级到"**判断用户写得对不对**"。

**1. 洪恩写字:从"动画笔顺"到"AI 智能评分 + 智能笔"**
洪恩 2023 年推出的《洪恩写字》app 配套**洪恩智能写字笔**(前端有微观摄像头 + 蓝牙),**实时捕捉笔画轨迹**,误差<0.1mm;**多维度 AI 智能评分**:从结构比例、重心平衡、笔画顺序、力度控制、形态美观等维度生成报告;**圈注纠错**:在屏幕上给写错的字标红、显示问题。

**2. 手写识别 API 的成熟**
百度 AI、好未来 AI、科大讯飞都在 2020 年前后开放了**手写中文识别 API**:
- 百度手写识别**准确率 >90%**
- 好未来 AI 开放平台"手写识别"产品页,对**试卷、作文、作业图片**做识别,字符准确率 90%+,手写英文 98%+

**3. 微信小程序 Canvas 描红的标准化方案**
根据 CSDN 2024-2025 年间多篇技术博客,小程序内做描红已基本收敛为如下技术栈:

```
WXML: <canvas type="2d" id="trace-canvas" bindtouchstart/move/end/>
JS:   const query = wx.createSelectorQuery()
      query.select('#trace-canvas').fields({node:true,size:true}).exec(res => {
        const canvas = res[0].node
        const dpr = wx.getSystemInfoSync().pixelRatio
        canvas.width = res[0].width * dpr
        canvas.height = res[0].height * dpr
        ctx.scale(dpr, dpr)
      })
```

**4. 微信小程序 + Hanzi Writer 的工程挑战**
典型坑点:
- **jsdelivr CDN 备案被撤销**(2021 年 12 月起),小程序不能直接 ajax 加载笔顺数据,需要用 `charDataLoader` 自定义加载;
- **wx.createCanvasContext 旧版 API 被废弃**,2.9.0 之后必须用 type="2d" + node 模式;
- **hanzi-writer-data 数据量太大**(9000+ 汉字),要么打包到本地(小程序包大小限制 20MB),要么用云函数按需加载。

### 2.6 纵向启示:对 shizi 项目的关键判断

**16 年的行业演进已经证明了一件事**:开源数据(Make Me a Hanzi)打平了技术门槛,**决定胜负的是"内容+体验"而非"笔顺精度"**。洪恩、有道能把 1300 字卖 300 元/年会员,靠的是游戏化 + 智能硬件 + 品牌;shizi 的差异化必须是"**更懂 3-6 岁认知节奏**",而非"更准的笔顺"。

但 —— "笔顺贴合字形"这件事,**没有任何一家做到字形数据与标准楷体字模像素级对齐**。这是行业的真空带,也是 shizi V2.4 改造的核心机会。

## 三、横向分析:主流产品对比与笔顺贴合度排名

### 3.1 市场格局

国内"儿童识字/练字"赛道已经形成清晰的两极分化:

**第一极(偏"识字"的综合产品,描红是子功能)**:洪恩识字(隶属 iHuman)、有道少儿词典(网易有道)、宝宝巴士汉字、叫叫识字、小伴龙、悟空识字。2024 Q3 洪恩系整体 MAU 创新高至 2912 万,同比 +14.8%;2022 Q1 单洪恩识字 1839 万 MAU、161 万付费用户。

**第二极(偏"练字"的硬件/课程产品,描红是核心)**:洪恩写字(2022 年 7 月发布,搭配洪恩 AI 智能写字笔)、河小象写字(2017 年创办,2019 年 B 轮融资 2 亿人民币,1500 万孩子)、麦田写字王(搭配麦田智绘神笔)。这一极走"AI 图像识别 + 笔顺比对"路线,核心壁垒是"识别引擎+纠错模型"而非"笔顺数据"本身。

**真正"笔顺贴合字体"做得好的产品,目前没有任何一家做到字形数据与标准楷体字模像素级对齐** —— 这是行业的真空带,也是 shizi V2.4 改造的核心机会。

### 3.2 各产品详细分析

#### 1. 洪恩识字(洪恩教育 NYSE:IH)

- **定位**:面向 2-8 岁的儿童识字 app(洪恩识字 1300 字);洪恩写字 6-12 岁,以软硬件一体(智能笔 + 定制写字本)主打硬笔书写练习。
- **笔顺技术**:**官方原话 ——"我们人工为 1300 个字、近一万个笔画进行编辑,只为让每个孩子写出的字都是不同的"**(云南网对洪恩识字制作人专访)。这意味着洪恩是**逐字人工编辑的路径数据**,而非基于开源库。1300 字 × 平均 8 笔 ≈ 1 万个笔画的编辑工作量,符合"5 年磨一剑"的产品节奏。
- **字体匹配度**:洪恩识字的"写"是**手指在屏幕上描**,孩子看到的是洪恩**自己绘制**的楷体范字 + 路径动画,**字形数据是洪恩美术团队原创**,而非调用系统字体。**字模与路径数据是同一组美术资产,贴合度天然就高**。
- **亮点 / 槽点**:
  - 亮点:4K 3D 场景,笔顺粒子特效,游戏化做的最重;"玩 → 认 → 练 → 写 → 测"五步闭环;艾宾浩斯 4 轮复习;护眼、防沉迷、家长端数据报告。
  - 槽点:游戏占比偏重,孩子容易沉迷;后期 298 元/年起价格不低。
- **对我们的启发**:
  - **洪恩"自研字形 + 自研路径"的一体化方案**是贴合度的天花板 —— 字怎么写和怎么画是同一批美术资产画出来的,不可能"对不上"。这是 V2.4 应该借鉴的核心理念:**字模、笔顺、动画三套数据要么用同一份源,要么用算法严格配准**。
  - 但 shizi 是小程序,不能像洪恩那样"5 年 1 万笔画人工编辑",需要依赖开源数据 + 后处理校准。

#### 2. 有道少儿词典(网易有道)

- **定位**:6-12 岁儿童语言学习工具,**主打"中文查字 + 英文查词"的词典 + 跟读 + 笔顺跟写**,免费无广告。
- **笔顺技术**:官方描述 ——"汉字全部配置笔顺动图及跟写练习,并为每一个汉字录入真人发音"。**所有 8000+ 通用规范汉字**均配置笔顺动图。技术实现推测:**SVG 渲染 + 笔顺数据(自研或合作版权)**。
- **字体匹配度**:笔顺跟写功能**有用户在小红书反馈"动图清晰、跟写对位准确"**。有道没有 AI 评分(免费版),跟写的"对错"反馈是**几何判定**(手指轨迹与预制 path 的距离)。
- **亮点 / 槽点**:
  - 亮点:收录《通用规范汉字表》全 8000 字(全行业最全);真人发音 + 自然拼读 + 跟读打分;无广告;内容权威(教育部 + 牛津小学词典合作)。
  - 槽点:定位是"工具"而非"课程",**没有游戏化设计**,孩子独立使用意愿低。
- **对我们的启发**:**有道证明了"全字库覆盖 + 笔顺动图"是儿童工具 app 的标配门槛**,shizi V2.4 之前 2256 字仅是字表(对应一级字表),但若想覆盖一二年级全部需求,需要扩展到至少 2500-3000 字。

#### 3. 宝宝巴士汉字(福州智永信息)

- **定位**:宝宝巴士是福州智永信息科技的产品矩阵,旗下有 200+ 款儿童 app,总下载量 8000 万家庭、月活千万级。宝宝巴士汉字是其**专为 2-8 岁汉字启蒙**的单品。
- **笔顺技术**:官方介绍 ——"**智能纠错笔画**,在线练字的时候可以自动纠正易错的笔画笔顺";"这里的每个汉字都有着对应的**书写顺序视频教程**"。推测实现:**预制视频(mp4/webm) + SVG 矢量动画混合**。
- **字体匹配度**:用户反馈"孩子可以在屏幕上跟着描写,它是会有语音+卡通图案水果或者汉堡这样的**笔顺指引方向**"。**宝宝巴士的最大优势是"低龄趣味"** —— 2-4 岁孩子不会挑剔字形是否严格规范。
- **亮点 / 槽点**:

## 四、技术深度:笔顺如何更贴合字体

### 4.1 问题定义

"笔顺贴合字体"在 shizi 这类描红 App 中实际包含 **4 个独立但相互关联的子问题**:

#### 1.1 引导线与字形轮廓的贴合
**理想状态**:浅色虚线/灰色"底字"与用户应该描写的字形 100% 重合。  
**现状痛点**:shizi 当前用 `ctx.fillText(char, ...)` 渲染底字(`pages/learn/learn.js:741-745`),系统字体(sans-serif)的字形轮廓与 `stroke-data.js` 中 Make Me a Hanzi 笔顺数据所基于的 **Arphic 楷体(Kaiti)** 不一致 → 虚线引导和"应该长成的样子"在视觉上就不重合。

#### 1.2 用户笔触与标准笔顺的比对
**理想状态**:用户用手指画出来后,系统能像 Hanzi Writer 的 `quiz()` 一样判断这一笔的"准确度"。  
**现状痛点**:shizi 的 `calcDeviation()`(`learn.js:937-967`)只算"点到折线段的最短距离",**没有考虑用户绘制的方向/起止点/笔顺/覆盖率**。评判阈值 35px(`learn.js:885`)过于粗糙,容易误判。

#### 1.3 笔画顺序的引导
**理想状态**:用户必须按 1→2→3 顺序写,系统依次高亮"当前该写的笔画"。  
**现状痛点**:shizi 的 `strokeIndex` 是手动 `++` 推进的(`completeCurrentStroke` 函数),用户即使写错顺序也会被接受(只要点离虚线够近)。

#### 1.4 错字检测与评分
**理想状态**:0-100 分,告诉用户"这字写得 87 分,捺笔方向反了"。  
**现状痛点**:shizi **完全没有量化评分**,只有"完成/再试一次"二元反馈。

### 4.2 主流技术方案

#### 方案 1:Hanzi Writer(主推)

- **是什么**:由 [chanind](https://github.com/chanind) 维护的开源 JS 库,GitHub 4.8k ⭐ 708 🍴,MIT License。最新 v3.7.3 发布于 2025-09-17,维护活跃。
- **数据源**:直接派生自 [Make Me a Hanzi](https://github.com/skishore/makemeahanzi)(Arphic 楷体 + Arphic PL UKai),字符数据存储在独立仓库 [hanzi-writer-data](https://github.com/chanind/hanzi-writer-data),每个字一个 JSON 文件。
- **核心能力**:
  1. **描边动画**:`writer.animateCharacter()` 按笔顺动态绘制每一笔
  2. **手写测验**:`writer.quiz()` 监听触摸事件,实时判定用户输入
  3. **错字检测**:返回 `onMistake(strokeData)` 回调,提供 `totalMistakes`、`mistakesOnStroke` 等统计
  4. **笔画顺序检测**:**强制**按 `quizStartStrokeNum` 顺序写,自动推进
  5. **笔画方向检测**:通过 medians(笔画中线)数据计算用户轨迹与标准方向的吻合度
  6. **提示机制**:`showHintAfterMisses: 3` 自动高亮该笔的引导
- **包大小**:
  - 核心 JS:**30 KB(压缩后 9 KB)**
  - 字符数据:每个汉字约 1-3 KB JSON,常用 3500 字 ≈ 2-5 MB(分文件加载,按需)
- **微信小程序兼容**:
  - 官方有插件版 [hanzi-writer-miniprogram](https://github.com/chanind/hanzi-writer-miniprogram),**但已于 2024-02-20 archive**(作者声明不再维护)
  - **替代方案**:把核心 JS 复制到项目本地,在小程序 `type="2d"` Canvas 上手动适配(代码改动量约 200-400 行)
- **集成到 shizi 的成本**:
  - 工作量:2-3 天(核心适配 + 数据打包 + 替换 drawStrokeGuide)
  - 风险:① 小程序 canvas 性能(每个字 1-3KB JSON 异步加载,首次有 100-300ms 延迟)② 1.6MB 现有 stroke-data.js **可保留作为 fallback**(数据格式兼容)
  - 收益:① 标准化笔顺 + ② 真正的错字检测 + ③ 评分 + ④ 字体完美贴合 + ⑤ 偏旁部首上色(教学亮点)

#### 方案 2:Make Me a Hanzi(数据源 · 重要!)

- **是什么**:**不是渲染库**,是 [skishore](https://github.com/skishore) 创建的**汉字数据仓库**,GitHub 2.5k ⭐ 590 🍴,46 个 commit,**数据已经稳定 5 年**。
- **数据规模**:**9000+** 常用汉字(简繁都有),数据文件:
  - `dictionary.txt`:每行一个 JSON,字段含 `character` / `pinyin` / `definition` / `decomposition`(IDS 描述) / `radical` / `matches`
  - `graphics.txt`:每行一个 JSON,含 `strokes`(SVG path 数组)和 `medians`(笔画中线坐标对)
- **数据格式(关键)**:
  ```json
  {
    "character": "我",
    "strokes": [
      "M 612.5 245.25 c 5.4 ... z",  // 第 1 笔的 SVG path
      "M 540 309 c 4.65 1.95 ... z",  // 第 2 笔
      ...
    ],
    "medians": [
      [[680, 250], [650, 270], [600, 290], ...],  // 第 1 笔中线点序列
      [[600, 320], [580, 340], ...],              // 第 2 笔中线点序列
      ...
    ]
  }
  ```
  - **坐标系**:`(0, 900)` 在左上角,`y` **向下递增**(与传统 SVG 相反!),需要 `transform="scale(1, -1) translate(0, -900)"` 翻转
  - **strokes** 是 SVG path(贝塞尔曲线)→ 渲染出来的形状 = **Arphic 楷体字形**
  - **medians** 是中线折线 → 可用于动画和描红引导
- **shizi 已经在用**:你的 `scripts/convert-stroke-data.js` 就是从这个数据源加工来的。这意味 shizi 的 1.6MB `stroke-data.js` **数据是 Make Me a Hanzi 的子集 + 二次加工**(3500 常用字)。
- **单独使用价值**:可以!如果你不想要 Hanzi Writer 库的渲染逻辑,可以**直接用 graphics.txt 的 strokes(SVG path)**,在 Canvas 里用 `Path2D` 描出来 → **字形 100% 贴合 Arphic 楷体**。

#### 方案 3:KanjiVG(日文备选 · 不推荐)

KanjiVG 是日文汉字(Kanji)的开源 SVG 矢量图,MIT 许可证,11000+ 字符。日式楷书(和"中文字帖"略有差异),简体字覆盖不全,**对 shizi 不建议直接用**。

#### 方案 4:自研方案(shizi 当前)

- **shizi 现在怎么做的**:
  1. `convert-stroke-data.js` 把 Make Me a Hanzi 的 medians 简化为 200×200 坐标系下的折线点序列
  2. `learn.js:741` 用 `ctx.fillText(self.data.currentChar, ...)` + `font: '140px sans-serif'` + `fillStyle: 'rgba(0, 0, 0, 0.06)'` 渲染"底字"
  3. `learn.js:784-818` 的 `drawStrokePath()` 用 `ctx.lineTo` 把点序列连成虚线(`setLineDash([8, 6])`)
  4. 用户触摸时,`calcDeviation()` 计算点到折线段距离
- **优点**:
  - 包小(JS 端只引 1.6MB `stroke-data.js`)
  - 简单(核心算法 50 行内)
  - 已有"完成/再试"反馈
- **缺点(关键)**:
  1. **字形不贴合**:底字用 `sans-serif` 系统字体,虚线用的是 Arphic 楷体数据 → 两者字形差异显著
  2. **140px 字号硬编码**:canvas 实际尺寸可能是 300×300 也可能是 600×600,140px 可能贴边
  3. **medians 简化过**:Douglas-Peucker 用 epsilon=3(1024 坐标系)简化为约 4-7 个点的折线 → 笔画的曲线美感丢失(楷书的弧度没了)
  4. **没有笔顺检测**:用户可以乱写,只要不离虚线太远就算过
  5. **没有评分**:`avgDeviation < 30` 的二元判定
  6. **SVG path 没用上**:`graphics.txt.strokes`(SVG path,带贝塞尔曲线)远比 medians(简化折线)更贴合字体

### 4.3 关键技术细节

#### 3.1 笔顺数据格式详解

| 格式 | 数据量/字 | 优点 | 缺点 | 代表项目 |
|---|---|---|---|---|
| **SVG path**(贝塞尔) | 1-3 KB | ① 100% 贴合 Arphic 楷体 ② 矢量缩放无损 ③ 视觉最美观 | ① 不容易做"中线距离"判定 ② 渲染需要 SVG 解析器 | Make Me a Hanzi `strokes` |
| **折线点序列**(medians) | 0.5-1 KB | ① 数据小 ② 容易做距离判定 ③ 适合描红引导 | ① 简化后丢失曲线 ② 视觉不贴合字体 | Make Me a Hanzi `medians` / **shizi 当前格式** |
| **二值掩码**(1024² 灰度图) | 128 KB+ | ① 训练 CNN 友好 ② 渲染最简单 | ① 巨量 ② 不支持单笔分离 | CCES 数据集 |
| **Unicode 编码**(笔画名序列) | 几十字节 | ① 极小 ② 跨字体 | ① 不知道坐标 ② 必须配合字体文件 | cnchar / Unicode IDS |

**最"贴合字体"的是 SVG path**。shizi 现在用的是 medians 简化版,所以视觉上有"折扣"。

#### 3.2 描红引导线的"贴合度"

**业界做法**:
1. **完全重合**(Hanzi Writer 默认):虚线 = 标准 medians,用户必须紧贴虚线写。**适合低龄初学者**。
2. **略有偏移**(各家描红本):底字浅色,用户在上面自由写,**只检查是否在"字形范围内"**。**适合有基础者**。
3. **shizi 当前**:底字 140px sans-serif + 虚线 medians → **两层不贴合**,所以"底字字形"和"该写的位置"对不上。

**建议方案**:
- **底字层用 SVG path 渲染**(从 graphics.txt 的 strokes 来)→ **底字和引导线 100% 贴合同一个字形**
- **虚线引导仍是 medians**(数据小,计算快)

#### 3.3 错字检测算法

| 方案 | 复杂度 | 准确度 | 适合 |
|---|---|---|---|
| **点对点距离** (shizi 当前) | O(n) | 60-70% | 初学者(粗糙判定) |
| **归一化 + DTW** | O(n²) | 80-90% | **推荐**:用户速度不同时也能正确匹配 |
| **双向图匹配** (学术方案) | O(n²) | 90%+ | 错字识别、书法评分 |
| **CNN / LSTM** | 训练成本高 | 95%+ | 商业 SDK(科大讯飞) |

**DTW(Dynamic Time Warping)实现示例**:
```javascript
function dtwDistance(userPath, stdPath) {
  const n = userPath.length, m = stdPath.length;
  const dtw = Array.from({length: n+1}, () => new Float64Array(m+1).fill(Infinity));
  dtw[0][0] = 0;
  for (let i = 1; i <= n; i++) {
    for (let j = 1; j <= m; j++) {
      const cost = Math.hypot(userPath[i-1].x - stdPath[j-1].x, 
                              userPath[i-1].y - stdPath[j-1].y);
      dtw[i][j] = cost + Math.min(dtw[i-1][j], dtw[i][j-1], dtw[i-1][j-1]);
    }
  }
  return dtw[n][m] / Math.max(n, m);  // 归一化
}
```

## 五、横纵交汇:shizi 的机会窗口

### 5.1 历史如何塑造了当下的竞争位置

把纵向的 16 年时间和横向的 8 个产品叠起来看,会出现一个清晰的故事弧:

**阶段一(2010-2014):封闭数据 + Flash 动画**
大厂(洪恩、悟空)用私有笔顺数据 + Flash 动画,孩子只能"看"不能"写"。**技术上没有"贴合"问题,因为根本没有用户笔触**。

**阶段二(2014-2017):开源数据 + Canvas 描红**
Make Me a Hanzi 2014 年发布,提供了开源的 9000+ 汉字笔顺数据。**但开发者用起来,普遍遇到的"贴合"问题,本质是"数据 + 字体 + 渲染"三者不同源**:
- 数据是 Arphic 楷体的 medians
- 字体是系统的 sans-serif
- 渲染是 Canvas 2D
三者各说各话。

**阶段三(2017-2020):小程序 + Hanzi Writer 出现**
Hanzi Writer 用 SVG 渲染 + 完整 medians,提供了"一站式"方案。**贴合问题本质解决了一部分** —— 动画跟数据同源了,但**用户的笔触怎么比对**?

**阶段四(2020-2026):AI 评分 + 智能笔**
洪恩、河小象等开始用 AI 模型 + 智能硬件做"评分"。**问题从"贴合"升级到"准确"** —— 不仅要动图对得上字形,还要判断用户写得像不像标准。

**shizi 现在的位置**:停在阶段二和阶段三之间 —— 数据用 Make Me a Hanzi 没问题,但渲染用系统字体 + Canvas 2D 简单绘制,**没有把"数据 / 字体 / 渲染 / 用户笔触"作为一个系统来设计**。

### 5.2 竞品的纵向对比

如果把主要竞品也放到时间线上看:

| 厂商 | 起点 | 路径 | 当前位置 |
|---|---|---|---|
| 洪恩 | 2003 年成立,2010 数字化 | 5 年人工编辑 + 自研数据 + 自研美术 | 1300 字质量天花板 |
| 有道 | 2017 年首代词典笔,2018 词典笔成系列 | 8 年迭代,2021 收 Hanzi Writer 数据 | 8000 字全字库 + 工具属性 |
| 宝宝巴士 | 2009 年成立,2010+ 启蒙 app | 重运营 + 多 app 矩阵 + 萌系美术 | 1440 字 + 趣味化 |
| 河小象 | 2017 年成立,2019 B 轮 2 亿 | 浙大书法老师 + AI 识别 + 真人点评 | 1500 万用户,质量稳健 |
| 叫叫 | 2015 年成立,2015 上线 | 故事化 IP 陪伴 | 识字不做写字(差异化定位) |
| **shizi** | 2024-2025 起步 | **Make Me a Hanzi + cnchar + 自研渲染** | **2256 字 + 微信生态 + 工具属性** |

**shizi 的起点比大厂晚 10-15 年**,但**借力开源生态**(Make Me a Hanzi + Hanzi Writer),**技术上能跨过早期 5 年的数据积累**。这是后发优势的体现。

### 5.3 优势的历史根源

shizi 现在的"自带优势",根源是 2014 年后开源生态的成熟:

1. **数据可用**:Make Me a Hanzi 提供 9000+ 汉字 SVG path + medians,**省去 5 年数据积累**。
2. **算法可用**:Hanzi Writer 提供完整手写测验算法,**省去 1 年 CV 算法研发**。
3. **生态可用**:cnchar 提供拼音/部首/笔画数,**省去工具库开发**。
4. **平台成熟**:微信小程序 Canvas 2D API 稳定,**省去平台适配**。

**这一切的根源是 2014-2017 年间那场"静悄悄的基础设施革命"**。shizi 是这场革命的红利接收方。

### 5.4 劣势的历史根源

shizi 当前的"贴合度只有 6.0 分"问题,根源有三条:

1. **数据简化损失**:`convert-stroke-data.js` 用 Douglas-Peucker 把 medians 简化成 4-7 个点的折线,**楷书弧度丢失**。这是工程上对包大小的妥协,但代价是视觉不贴合。
2. **字体替换损失**:`learn.js:741` 用 `sans-serif` 渲染底字,**系统字体跟 Arphic 楷体不是一套**。这是"用 canvas 填文字"时的快速做法,代价是字形对不上。
3. **无 AI 评分**:`calcDeviation()` 只算"点到线段距离",**没有时间维度对齐**(DTW)、没有方向检测、没有覆盖率判定。代价是反馈粗糙。

**这三条都是"为了快速上线"的妥协**,但**今天回头看,正是改造的最佳切入点**。

### 5.5 未来推演:三个剧本

#### 剧本 A:最可能的(70% 概率)

**V2.4 改造聚焦"轻量级贴合"**:
- 底字层从系统字体换成 SVG path(从 graphics.txt 来)
- DTW 评分上线
- 数据从 1.6MB 扩展到 3MB,放进分包
- 总工程量 1-2 周

**结果**:贴合度从 6.0 → 7.5 分(逼近有道水平),用户感知"字写起来跟动画对得上了"。**这个剧本下,shizi 不会立刻获得显著增长,但会稳住"工具型"用户**。

#### 剧本 B:最危险的(20% 概率)

**V2.4 改造走过头,引入 AI 评分 / 智能笔**:
- 学洪恩做智能笔
- 学河小象做 AI 识别
- 包大小失控(5MB+)
- 开发周期失控(2-3 个月)
- 智能笔硬件合作方扯皮

**结果**:技术债累积,产品迭代慢,用户增长停滞在"工具"定位。**这个剧本下,shizi 可能被洪恩系用户群边缘化**(他们会升级到洪恩写字)。

#### 剧本 C:最乐观的(10% 概率)

**V2.4 改造做出"微信生态独有的卖点"**:
- 利用微信生态(小程序码、社群分享)做"家长拼团"
- 描红评分自动生成"今日学习报告"发到家长微信
- 跟企业微信合作,做"老师布置作业"场景
- 切入 5-12 岁市场(目前是 3-6 岁)

**结果**:从"个人工具"升级为"教学系统",**有可能**实现 3-5 倍用户增长。但前提是产品先做到 7.5+ 分贴合度(剧本 A),否则体验撑不住企业级需求。

### 5.6 文化升维:从"练字"到"汉字教育"

回头看,这个赛道的本质,不是"做一个练字 app",而是"用数字工具传承汉字文化"。

2010 年,家长焦虑的是"孩子能不能认字";2026 年,家长焦虑的是"孩子能不能写对字"。**焦虑的迁移,反映了教育需求的升级**。洪恩的"4K 3D"、河小象的"1V1 真人点评"、有道的"全字库 + 真人发音",都是对这种焦虑的回应。

但**焦虑的本质没变**:家长希望孩子写出来的字"像样"。**这是任何技术方案、任何商业模式、任何 IP 包装都无法绕过的底层需求**。

shizi V2.4 改造的本质,不是追新潮、不是卷技术、不是拼 AI 大模型。**而是回到"字写得对不对"这个最朴素的问题上,把答案做得更精确**。这是独立开发者相对大厂的核心壁垒 —— **不拼运营、不拼资本、不拼噱头,只拼对"字形"本身的尊重**。

## 六、行动计划:V2.4 改造的 3 个具体方案

### 方案 A:引入 Hanzi Writer(激进 · 收益最大)

| 维度 | 评估 |
|---|---|
| **工作量** | **2-3 天** |
|  | Day 1: 把 hanzi-writer 核心 JS 复制到 `utils/`,移植 medians loader |
|  | Day 2: 在 `learn.js` 替换 `drawStrokeGuide()` 为 Hanzi Writer 的 quiz 模式 |
|  | Day 3: 适配小程序 Canvas 2D(type="2d",修复 onTouchMove 事件差异) |
| **风险** | ① hanzi-writer-miniprogram 已 archive,需自适配 ② 数据文件需要分包(主包放 3500 字,其他字动态加载) ③ 异步加载有 100-300ms 首次延迟 |
| **收益** | ① 标准化笔顺 ② 真正的错字检测 ③ 完整评分机制 ④ 字体完美贴合 ⑤ 偏旁部首上色(教学亮点) ⑥ 与开源社区同步更新 |
| **包大小影响** | 核心 JS +30KB(可接受);字符数据建议分包,常用 3500 字 ≈ 3MB(分包限制 20MB 内) |
| **可商用性** | ✅ MIT + Arphic,完全可商用 |

**推荐指数**:⭐⭐⭐⭐⭐ 如果你想做"完整练字 App"

### 方案 B:自研渲染 + 改用 SVG path 数据(推荐 · 平衡)

| 维度 | 评估 |
|---|---|
| **工作量** | **1-2 天** |
|  | Step 1(2h): 修改 `convert-stroke-data.js`,输出**两种**格式:① 保留 medians(虚线引导)② 额外输出 `strokes`(SVG path,用于底字) |
|  | Step 2(2h): `drawStrokeGuide()` 中底字层用 `Path2D` + `ctx.fill()` 渲染 SVG path,颜色浅灰 |
|  | Step 3(2h): 实现简单的 DTW 评分(用上面给的代码) |
|  | Step 4(2h): 加入"方向检测"(判断用户触摸轨迹的"起点→终点"方向) |
| **风险** | ① 微信小程序 Canvas 2D 支持 `Path2D`(需基础库 2.10.0+,占比 99%+) ② 字符数据翻倍(~3MB,但仍可控) |
| **收益** | ① 底字和引导线 **100% 贴合同一字形** ② 真实评分 ③ 方向检测 ④ 包可控 |
| **包大小影响** | 字符数据从 1.6MB → ~3MB(可放分包) |

**推荐指数**:⭐⭐⭐⭐⭐ **这是最适合 shizi 的方案** —— 改动小、风险低、收益大

**关键代码片段**(改 `drawStrokeGuide`):
```javascript
// 加载数据
var strokeInfo = StrokeData.getStrokeData(currentChar);
// strokeInfo.strokes 既有 medians(point 数组)又有 svgPath(字符串)

drawStrokeGuide: function() {
  var strokeInfo = StrokeData.getStrokeData(self.data.currentChar);
  var scale = canvasWidth / 1024;  // SVG path 是 1024 坐标系
  
  // ① 底字层:用 SVG path 描浅灰(关键!贴合字形)
  ctx.fillStyle = 'rgba(0, 0, 0, 0.06)';
  strokeInfo.strokes.forEach(function(s) {
    if (s.svgPath) {
      var path2d = new Path2D(s.svgPath);
      // 1024→canvas 缩放,且翻转 Y 轴
      ctx.save();
      ctx.translate(0, canvasHeight);
      ctx.scale(scale, -scale);
      ctx.fill(path2d);
      ctx.restore();
    }
  });
  
  // ② 虚线引导:仍是 medians(数据小,实时绘制快)
  var medianPoints = strokeInfo.strokes[strokeIndex].points;
  ctx.setLineDash([8, 6]);
  ctx.strokeStyle = '#4A90D9';
  ctx.lineWidth = 4;
  ctx.beginPath();
  ctx.moveTo(medianPoints[0].x * scaleX, medianPoints[0].y * scaleY);
  medianPoints.forEach((p, i) => i > 0 && ctx.lineTo(p.x * scaleX, p.y * scaleY));
  ctx.stroke();
}
```

### 方案 C:维持现状 + 局部优化(保守)

| 维度 | 评估 |
|---|---|
| **工作量** | **半天** |
|  | ① 字号自适应(140px → `Math.min(W,H) * 0.7`) ② 字体改成 "楷体/Kaiti" ③ 把虚线宽度增加 1-2px 提高可见性 |
| **风险** | 低(纯样式调整) |
| **收益** | 视觉改善 30%,但**字形贴合度仍差**(系统楷体和 Arphic 楷体还有差异) |

**推荐指数**:⭐⭐⭐(如果短期内没空,先做这个)

### 我的建议

**先做方案 B 的 Step 1+2**(2-4 小时),看效果;如果数据从 1.6MB → 3MB 可以接受,**继续做 Step 3+4**。如果包大小接受不了,**回退到方案 C**。

理由:
- 方案 B 的核心收益(底字和引导线 100% 贴合)是肉眼可见的"飞跃",**单这一步就能把贴合度从 6.0 提到 7.0**。
- DTW 评分是锦上添花,不是必做。
- 方案 A(引入 Hanzi Writer)是长期目标,但需要先验证方案 B 的基础架构能跑通。
