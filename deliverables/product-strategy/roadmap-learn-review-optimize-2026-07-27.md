## 📌 TL;DR

本路线图将识字小程序学习复习模块优化分为 **V2.1 / V2.2 / V2.3** 三个版本递进交付，总计 17 人天（前端1人+后端1人），17个自然日完成。V2.1 紧急修复数据信任危机（1人天），V2.2 建立间隔重复与递进教学核心机制（6人天），V2.3 提升体验与深度（10人天）。灰度策略从 5% 逐步扩展至 100%，每版本配备独立回滚方案。核心风险集中在数据迁移（旧版假数据约56%假阳性）和 ASR 降级策略，已制定专项应对方案。

## 1. 当前指标表现（来自数析）

| 指标 | 当前值 | 行业基准 | 差距 |
|------|--------|---------|------|
| 综合假阳性率 | ~56% | <5% | 严重偏离，数据不可信 |
| 30天知识保持率 | ~10% | 60-70%（古古识字92%） | 严重不足 |
| 新字学习完成率 | ~70% | 85-90% | 流程挫败感导致退出 |
| 7日留存率 | ~40% | 55-65% | 缺乏复习机制导致遗忘流失 |
| 学习过程退出率 | ~30% | <15% | 直接跟读步骤导致挫败 |

**数析核心判断**：当前系统最严重的问题是 **数据信任崩溃** — 56%假阳性意味着超过一半的"掌握"判定可能是假的，所有基于此数据的分析、报告和决策都不可信。必须首先修复数据源头，否则后续所有优化都建立在沙上。

## 2. 用户反馈摘要（来自瑞思）

| 痛点 | 严重度 | 受影响用户 | 根因 |
|------|--------|-----------|------|
| "学完就忘" | 极高 | 几乎所有用户 | 无间隔重复，30天保持率仅10% |
| "标记掌握但实际不认识" | 极高 | 家长用户 | Math.random()判定导致56%假阳性 |
| "新字直接跟读太难" | 高 | 3-4岁用户 | 违背"再认先于回忆"认知规律 |
| "反复出错没有帮助" | 中 | 学习困难用户 | 错误后直接给答案，无渐进提示 |
| "不知道一天该学多少" | 中 | 家长用户 | 无每日学习量控制 |

**瑞思核心判断**：3-6岁儿童工作记忆容量仅3-5项、注意力时长5-8分钟，当前2步学习流程（展示+跟读）严重超出认知负荷。四步递进（再认→回忆→动觉→产出）是降低认知负荷的关键，而非增加负担。

## 3. 竞品动态（来自竞析）

| 竞品 | 关键动态 | 对我们的启示 |
|------|---------|-------------|
| 古古识字 | 唯一采用间隔重复+渐进提示，3月留存92% | 间隔重复是留存的核心杠杆 |
| 洪恩识字 | 游戏化极强，但200字后效率下降 | 游戏化有拐点，服务于学习而非替代 |
| 叫叫识字 | 4步递进+3字小复习 | 小复习是稳定学习效果的关键环节 |
| 行业趋势 | 递进教学已成标配（4-7步），我们仅2步严重落后 | 必须尽快补齐递进教学 |

**竞析核心判断**：我们在教学步骤数（2步vs行业4-7步）和间隔重复（无vs古古识字有）两个维度严重落后，但微信生态（推送提醒+社交分享）是8款竞品均不具备的独有优势。

#### V2.1 — 数据修复（Day 1）

**主题**：止血 — 消除假阳性，恢复数据信任

- [ ] learn.js L275/280/287处代码不再包含随机 fallback 给分
- [ ] 云函数 L668-683处 ASR 失败走降级逻辑（标记"待判定"）
- [ ] 所有"待判定"记录在管理后台可查询
- [ ] 修复后综合假阳性率 < 5%（灰度5%验证3小时）

- Day 1 14:00 灰度5%，观察2小时

- **Feature Flag 控制**：通过云函数配置项控制是否启用新学习流程和新复习算法

- **回滚决策阈值**：
  - 20%灰度用户中新流程完成率 < 60%（低于旧流程70%）
  - recordReview闭环错误率 > 5%
  - 数据迁移导致用户可见的学习进度丢失 > 0.1%

**风险与应对**：

## 8. 数据迁移方案

### 8.1 迁移范围

| 数据源 | 当前结构 | 目标结构 | 记录量级（估计） |
|--------|---------|---------|----------------|
| learning_progress | openid, char_id, status, correct_count, wrong_count, last_review_date | 新增 box_level, next_review_date, review_interval, first_learn_date, last_correct_date, consecutive_correct, consecutive_wrong, recognition_correct, recall_correct, cross_day_correct, error_type, error_count_by_type, is_assisted | ~10万条 |
| mastered_chars | openid, char_ids[] | 映射到 learning_progress，status='familiar' | ~5千条 |

### 8.2 迁移策略

**核心原则**：保守映射、不丢数据、可逆可验证

#### 策略一：learning_progress 增量字段填充

```
旧字段保留不变，新增字段按以下规则填充：

box_level 映射规则：
  - 旧 status='mastered' 且 correct_count >= 5 → box_level = 3 (familiar, 非mastered!)
  - 旧 status='mastered' 且 correct_count < 5 → box_level = 2 (seeing)
  - 旧 status='learning' → box_level = 1 (seeing)
  - 旧 status='new' → box_level = 1 (new)
  
  ⚠️ 关键决策（Q-2）：旧"掌握"映射为"familiar"而非"mastered"
  原因：旧"掌握"含56%假阳性，直接映射为"mastered"会污染新系统
  "familiar"意味着"曾学习过但需验证"，通过后续间隔复习自然晋升

next_review_date 映射规则：
  - box_level = 1 → next_review_date = 今天（明天需复习）
  - box_level = 2 → next_review_date = 今天 + 1天
  - box_level = 3 → next_review_date = 今天 + 3天
  
review_interval 映射规则：
  - box_level = 1 → 1
  - box_level = 2 → 3
  - box_level = 3 → 7

first_learn_date 映射规则：
  - 无旧数据 → last_review_date - 7天（估算）

last_correct_date 映射规则：
  - 使用 last_review_date（最接近的值）

consecutive_correct / consecutive_wrong：
  - 默认 0（无法从旧数据推算）

recognition_correct / recall_correct / cross_day_correct：
  - 默认 0（旧数据不区分验证维度）
```

#### 策略二：mastered_chars 拆分映射

```
mastered_chars.char_ids[] → 拆分为 learning_progress 多条记录

处理逻辑：
1. 遍历每个 openid 的 mastered_chars.char_ids
2. 对每个 char_id：
   - 若 learning_progress 中已有该 openid+char_id 的记录 → 更新（不覆盖已有数据）
   - 若无记录 → 新建，status='familiar', box_level=3, next_review_date=今天+3天
3. 标记原 mastered_chars 记录为 "migrated: true"
```

#### 策略三：review_logs 假数据标记

```
识别 Math.random() 产生的假数据：
1. 筛选 review_logs 中 question_type='say_word' 的记录
2. 筛选 created_at < V2.1修复时间 的记录
3. 这些记录的 result 字段不可信
4. 标记为 "data_quality: "unreliable_pre_fix""

处理方式：
- 不删除（保留审计轨迹）
- 不计入历史正确率统计
- learning_progress 的 correct_count/wrong_count 不基于这些记录重算
- 仅在 review_logs 聚合分析时排除这些记录
```

### 8.3 迁移执行方案

```
迁移分3批执行，每批间隔2小时验证：

批次1（Day 2 10:00）：7天内活跃用户
  - 预计影响：~30% 记录
  - 验证：随机抽查50条，检查字段映射正确性
  - 回滚：如发现错误，停止后续批次，修正映射规则

批次2（Day 2 14:00）：30天内活跃用户
  - 预计影响：~60% 记录
  - 验证：抽查100条 + 与批次1数据交叉验证

批次3（Day 2 18:00）：全量用户
  - 预计影响：~10% 剩余记录
  - 验证：全量数据一致性检查
```

### 8.4 迁移验证清单

- [ ] 迁移前后记录总数一致（无丢失）
- [ ] 旧字段值保持不变（不破坏现有逻辑）
- [ ] 新字段映射规则正确（抽查200条验证）
- [ ] 索引创建成功（查询性能验证）
- [ ] mastered_chars 标记 migrated=true
- [ ] review_logs 假数据标记完成
- [ ] 灰度5%用户迁移后学习进度无异常
- [ ] 回滚脚本可用（5分钟内可执行）

### 8.5 迁移回滚方案

```
迁移回滚（极端情况）：

1. 删除迁移新增字段：云函数批量移除 box_level, next_review_date 等新字段
   - 耗时：~30分钟（10万条记录）
   - 影响：回退到V2.1状态，新功能不可用

2. 恢复 mastered_chars.migrated 标记：批量设为 false
   - 耗时：~5分钟

3. 清理 review_logs 假数据标记：批量移除 data_quality 字段
   - 耗时：~10分钟

⚠️ 注意：迁移回滚是最后手段，因为迁移期间用户可能已经基于新字段产生了新数据
```
